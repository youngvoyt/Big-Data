## 4. Подготовка данных (ETL)

На этапе ETL выполнялась очистка исходного набора данных и отбор наблюдений, пригодных для решения задачи.

### 4.1 Выполненные действия

1. JSON-файл был загружен в `Spark DataFrame`.
2. Была проанализирована схема данных.
3. Проверено наличие пропусков в ключевых полях.
4. Из выборки удалены строки:
   - с `gender`, отличным от `male` и `female`;
   - с отсутствующим `speed`;
   - с массивом `speed`, содержащим слишком мало точек.

### 4.2 Логика фильтрации

Для корректного вычисления статистик скорости необходимо, чтобы:

- массив `speed` существовал;
- содержал достаточное число наблюдений;
- целевая переменная была корректно определена.

### 4.3 Фрагмент кода очистки

```python
df_clean = (
    raw_df
    .filter(F.col("gender").isin("male", "female"))
    .filter(F.col("speed").isNotNull())

```

## 5. Конструирование признаков (Feature Engineering)

Основная особенность варианта 10 состоит в том, что нужные признаки отсутствуют в готовом виде и должны быть рассчитаны из массива `speed`.

### 5.1 Построенные признаки

В работе использованы следующие признаки:

- `avg_speed` — среднее арифметическое значений массива `speed`;
- `stddev_speed` — стандартное отклонение значений массива `speed`;
- `speed_len` — длина массива скорости, добавленная как вспомогательный числовой признак.

### 5.2 Смысл признаков

**`avg_speed`** позволяет оценить средний темп тренировки.  
**`stddev_speed`** показывает, насколько скорость менялась по ходу тренировки:

- низкое значение соответствует более ровному темпу;
- высокое значение соответствует более нестабильному темпу.

Таким образом, признак `stddev_speed` напрямую отражает интересующий нас показатель — **стабильность темпа**.

### 5.3 Способ вычисления

Поскольку данные представлены в виде массивов, признаки рассчитывались с помощью встроенных функций Spark SQL:

- `aggregate`
- `transform`
- `size`
- `sqrt`

### 5.4 Пример кода построения признаков

```python
mean_expr = "aggregate(speed, cast(0.0 as double), (acc, x) -> acc + x) / size(speed)"

var_expr = f"""
aggregate(
    transform(speed, x -> pow(x - ({mean_expr}), 2)),
    cast(0.0 as double),
    (acc, x) -> acc + x
) / (size(speed) - 1)
"""

df_features = (
    df_clean
    .select(
        "gender",
        F.size("speed").alias("speed_len"),
        F.expr(mean_expr).alias("avg_speed"),
        F.when(F.size("speed") > 1, F.sqrt(F.expr(var_expr)))
         .otherwise(F.lit(0.0))
         .alias("stddev_speed")
    )
)
```

### 5.5 Значение этапа Feature Engineering

Именно на этом этапе сырые массивные данные преобразуются в формат, пригодный для машинного обучения.  
Без такого преобразования `Spark MLlib` не может использовать данные напрямую.

## 6. Построение ML-пайплайна

Для выполнения требований лабораторной работы использован **Pipeline API** из библиотеки `pyspark.ml`.

### 6.1 Состав пайплайна

В пайплайн вошли следующие этапы:

1. `StringIndexer`  
   Кодирование строковой переменной `gender` в числовой `label`.

2. `VectorAssembler`  
   Объединение признаков `avg_speed`, `stddev_speed`, `speed_len` в вектор `features`.

3. `StandardScaler`  
   Масштабирование признаков перед обучением модели.

4. `LogisticRegression`  
   Построение бинарного классификатора с регуляризацией `ElasticNet`.

### 6.2 Почему выбран именно Pipeline

Использование `Pipeline` удобно по нескольким причинам:

- обеспечивает воспроизводимость обработки;
- объединяет preprocessing и model fitting в единый объект;
- соответствует промышленной практике;
- прямо требуется методическими указаниями задания.

### 6.3 Настройка модели

Использовалась следующая конфигурация логистической регрессии:

```python
lr = LogisticRegression(
    labelCol="label",
    featuresCol="scaled_features",
    maxIter=40,
    regParam=0.15,
    elasticNetParam=0.5
)
```

### 6.4 Роль ElasticNet

Регуляризация ElasticNet сочетает свойства:

- `L1`-регуляризации — способствует разреженности и помогает выделять важные признаки;
- `L2`-регуляризации — повышает устойчивость модели и уменьшает переобучение.

Для данной задачи это особенно удобно, так как модель остается интерпретируемой и при этом лучше контролирует шум.

## 8. Оценка качества модели

Для анализа качества были использованы следующие метрики:

- `ROC-AUC`
- `Accuracy`
- `Confusion Matrix`

### 8.1 Почему именно эти метрики

**ROC-AUC** показывает, насколько хорошо модель разделяет два класса по вероятностям.  
**Accuracy** показывает долю правильных ответов.  
**Confusion Matrix** позволяет увидеть структуру ошибок, то есть понять, какой из классов модель путает чаще.

### 8.2 Интерпретация результата

После запуска ноутбука в Colab в отчет следует перенести фактические значения:

- `ROC-AUC = ...`
- `Accuracy = ...`
- `Confusion Matrix = ...`

### 8.3 Как интерпретировать значения

- если `ROC-AUC` заметно выше 0.5, модель действительно извлекает полезный сигнал из данных;
- если `Accuracy` высока, модель правильно классифицирует значительную долю тренировок;
- если матрица ошибок показывает сильный перекос, это может означать влияние дисбаланса классов.

### 8.4 Важное замечание

Так как выборка по полу не идеально сбалансирована, интерпретировать `Accuracy` следует осторожно.  
Поэтому в работе акцент сделан прежде всего на **ROC-AUC** и анализе коэффициентов модели.

## 9. Интерпретация коэффициентов модели

Одним из главных достоинств логистической регрессии является возможность интерпретации коэффициентов.

### 9.1 Общий принцип

Если коэффициент:

- **положительный**, рост признака повышает вероятность положительного класса;
- **отрицательный**, рост признака уменьшает вероятность положительного класса;
- **близок к нулю**, влияние признака на решение модели невелико.

### 9.2 Ключевой признак — `stddev_speed`

Для варианта 10 центральный интерес представляет именно `stddev_speed`, поскольку он отражает стабильность темпа.

Возможны три интерпретации:

1. **Положительный коэффициент**  
   Более высокая вариативность темпа сильнее связана с положительным классом.

2. **Отрицательный коэффициент**  
   Более стабильный темп сильнее связан с положительным классом.

3. **Коэффициент около нуля**  
   Стабильность темпа сама по себе не является сильным предиктором пола.

### 9.3 Аналитический смысл

Здесь важно подчеркнуть, что модель не выявляет “биологический закон”, а только находит **статистический паттерн в конкретных исторических данных**.  
Поэтому интерпретация должна быть аккуратной и использоваться только как вспомогательная аналитика.

## 10. Бизнес-интерпретация

С прикладной точки зрения такая модель полезна не как инструмент жесткого определения пола пользователя, а как дополнительный поведенческий сигнал.

### 10.1 Возможные сценарии применения

1. **Персонализация предложений**  
   Если профиль пользователя заполнен частично, модель может помочь выбрать более релевантные рекомендации.

2. **Таргетированная реклама**  
   Сегментация по поведенческим паттернам помогает снизить количество нерелевантных рекламных показов.

3. **CRM и удержание пользователей**  
   Дополнительные характеристики пользовательского профиля позволяют строить более точные коммуникационные сценарии.

4. **Рекомендательные системы**  
   Поведенческие признаки тренировки могут использоваться для подбора более подходящих программ и сценариев занятий.

### 10.2 Ответ на вопрос варианта

Связь между стабильностью темпа и полом пользователя определяется знаком и величиной коэффициента при `stddev_speed`.

- если коэффициент заметен по модулю, то стабильность темпа действительно участвует в предсказании;
- если коэффициент слабый, то этот признак нельзя считать самостоятельным надежным индикатором.

Следовательно, **стабильность темпа следует использовать не изолированно, а как один из нескольких поведенческих признаков**.
