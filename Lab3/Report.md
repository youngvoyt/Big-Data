## 1. Введение

Цель работы состояла в построении сквозного ML-пайплайна на Apache Spark для решения прикладной задачи фитнес-платформы HealthTech.  
Для варианта 10 требовалось предсказать пол пользователя (`gender`) по параметрам тренировки, используя признаки:

## 2. Предобработка данных

В качестве исходного набора использован `endomondoHR.json`. Полный файл имеет очень большой размер, поэтому для запуска в Colab была подготовлена облегченная версия `endomondoHR_sample_v10.json.gz`.

На этапе ETL выполнялись следующие действия:

1. Загрузка JSON-файла в Spark DataFrame.
2. Анализ схемы данных и проверка наличия пропусков.
3. Фильтрация строк:
   только записи с `gender = male/female`;
   только записи, где массив `speed` не пуст и содержит достаточное число точек.
4. Удаление аномально коротких тренировок.

Пример логики очистки:

```python
df_clean = (
    raw_df
    .filter(F.col("gender").isin("male", "female"))
    .filter(F.col("speed").isNotNull())
    .filter(F.size(F.col("speed")) >= 10)
)
```

Такой подход позволил оставить только те наблюдения, где признаки темпа действительно информативны.

## 3. Конструирование признаков

Для варианта 10 были построены признаки:

- `avg_speed` — среднее значение по массиву `speed`
- `stddev_speed` — стандартное отклонение значений в массиве `speed`
- `speed_len` — длина массива скорости как дополнительный технический признак, отражающий длину записи

Пример кода:

```python
df_features = df_clean.select(
    "gender",
    F.size("speed").alias("speed_len"),
    F.expr("aggregate(speed, cast(0.0 as double), (acc, x) -> acc + x) / size(speed)").alias("avg_speed"),
    F.when(
        F.size("speed") > 1,
        F.sqrt(
            F.expr("""
                aggregate(
                    transform(
                        speed,
                        x -> pow(x - (aggregate(speed, cast(0.0 as double), (acc, y) -> acc + y) / size(speed)), 2)
                    ),
                    cast(0.0 as double),
                    (acc, x) -> acc + x
                ) / (size(speed) - 1)
            """)
        )
    ).otherwise(F.lit(0.0)).alias("stddev_speed")
)
```

Для подготовки данных к модели использован Pipeline API:

- `StringIndexer` для кодирования целевой переменной
- `VectorAssembler` для объединения признаков
- `StandardScaler` для нормализации
- `LogisticRegression` с параметрами `regParam` и `elasticNetParam`

## 6. Интерпретация коэффициентов

Ключевая аналитическая часть варианта 10 — анализ коэффициентов логистической регрессии.

Если коэффициент:

- положительный, то рост признака увеличивает вероятность положительного класса
- отрицательный, то рост признака уменьшает вероятность положительного класса

Особый интерес представляет `stddev_speed`:

- если коэффициент по модулю заметен, значит стабильность темпа действительно несет сигнал о поле пользователя
- если коэффициент близок к нулю, значит один только показатель стабильности темпа недостаточно информативен

## 7. Бизнес-вывод

Даже если модель не предназначена для жесткого автоматического определения пола, она полезна как вспомогательный инструмент в нескольких сценариях:

1. Персонализация предложений.  
   Если профиль пользователя заполнен не полностью, модель помогает мягко уточнить сегмент для рекомендаций товаров и тренировочных планов.

2. Улучшение таргетинга.  
   Более точное понимание пользовательского профиля повышает релевантность рекламных кампаний и уменьшает долю нецелевых показов.

3. Повышение качества CRM-аналитики.  
   Даже вероятностная оценка пола может использоваться вместе с другими сигналами, а не как единственный источник истины.

С точки зрения продукта стабильность темпа может быть косвенным индикатором стиля тренировки. Это не биологическая закономерность сама по себе, а паттерн поведения в исторических данных. Поэтому бизнесу важно использовать такую модель осторожно и только как часть более широкой системы персонализации.
